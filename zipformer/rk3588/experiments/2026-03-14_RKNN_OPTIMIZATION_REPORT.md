# RK3588 NPU 음성인식 속도 최적화 보고서

**작성일:** 2026-03-14
**작성자:** nsbb
**프로젝트:** 한국어 실시간 음성인식(STT)을 RK3588 NPU에서 빠르게 돌리기
**목표:** encoder 추론 시간을 33ms → 20ms로 줄이기 (성능 유지)

---

## 1. 이 문서가 다루는 것

RK3588이라는 ARM 칩에 달린 **NPU(Neural Processing Unit, 인공지능 전용 프로세서)**에서 한국어 음성인식 AI 모델을 돌리고 있다. 이 AI 모델은 사람의 말을 텍스트로 바꿔주는 일을 한다.

현재 문제: **AI가 생각하는 데 35ms(0.035초)가 걸린다.** 목표는 이걸 **20ms(0.02초)**로 줄이는 것이다.

이 보고서는 12가지 서로 다른 최적화 방법을 시도한 전체 과정과 결과를 기록한다.

---

## 2. 배경 지식 — 쉬운 설명

### 2.1 음성인식이 작동하는 방식

```
마이크 → [소리를 숫자로 변환] → [AI 모델이 분석] → "안녕하세요" (텍스트)
         ~~~~~~~~~~~~~~~~~~~    ~~~~~~~~~~~~~~~~
         fbank (전처리)          encoder (핵심, 여기가 느림)
```

사람이 말하면, 소리가 숫자 배열(fbank)로 변환되고, 이 숫자를 AI 모델(encoder)이 받아서 "이건 '안녕하세요'라는 뜻이다"라고 판단한다.

### 2.2 "스트리밍" 이란?

음성인식에는 두 가지 방식이 있다:

- **오프라인:** 말이 다 끝날 때까지 기다렸다가 한 번에 처리. 유튜브 자막 같은 것.
- **스트리밍:** 말하는 도중에 실시간으로 처리. 시리, 구글 어시스턴트 같은 것.

우리가 만드는 건 **스트리밍** 방식이다. 소리를 **0.32초(32프레임)짜리 조각(chunk)**으로 잘라서 매 조각마다 AI가 처리한다. 따라서 AI가 0.32초 안에 한 조각을 처리하지 못하면 실시간이 안 된다.

현재 35ms면 여유가 있긴 하지만(0.32초 = 320ms이니까), 다른 처리도 해야 하므로 빠를수록 좋다.

### 2.3 NPU란?

| 프로세서 | 역할 | 특기 |
|---------|------|------|
| CPU | 일반 프로그램 실행 | 뭐든 할 수 있지만 AI 연산은 느림 |
| GPU | 그래픽/병렬 연산 | 게임, AI 학습 |
| **NPU** | **AI 전용 연산** | **행렬 곱셈에 특화, 전력 효율 높음** |

RK3588 칩에는 NPU가 내장되어 있다. 이걸 잘 쓰면 CPU보다 빠르고 전력도 적게 쓸 수 있다. 하지만 **NPU한테 일을 시키려면 데이터를 NPU 전용 메모리로 보내야 하고, NPU가 이해할 수 있는 형태로 모델을 변환해야 한다.** 이 과정에서 오버헤드(추가 시간)가 발생한다.

### 2.4 RKNN이란?

**RKNN = Rockchip Neural Network**. RK3588 NPU를 사용하기 위한 소프트웨어 도구 모음이다.

```
[ONNX 모델] → [RKNN 변환기] → [RKNN 모델] → [NPU에서 실행]
  (범용)       (rockchip 도구)   (NPU 전용)    (빠르게 추론)
```

### 2.5 핵심 숫자들

| 항목 | 값 | 의미 |
|------|-----|------|
| 모델 크기 | 69M 파라미터 | 6900만 개의 학습된 숫자들 |
| ONNX 노드 수 | 2,275개 | AI 모델 안의 계산 단계 수 |
| RKNN 레이어 수 | 1,619개 | NPU가 실행하는 단계 수 |
| 입력 텐서 수 | 36개 | 매 chunk마다 NPU에 보내는 데이터 묶음 |
| 입력 데이터 크기 | 2.0 MB | 매 chunk마다 전송되는 데이터량 |
| 현재 추론 시간 | ~35ms | 한 chunk 처리에 걸리는 시간 |
| 목표 추론 시간 | 20ms | 달성하려는 시간 |

## 3. 병목 진단 — 왜 느린가?

### 3.1 시간이 어디서 소비되는가

```
전체 35ms의 구성:

┌─────────────────────────────────────────────────────────────┐
│  write_input (데이터를 NPU로 보내기)                    0.9ms │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  rknn_run (NPU 실행) ██████████████████████████████████ 31ms │
│                                                             │
│    ├─ 실제 계산 (행렬 곱셈 등)              ~2ms            │
│    └─ 레이어 전환 오버헤드 (dispatch)       ~29ms  ← 병목!  │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  read_output + cache 변환                               1.2ms │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 "레이어 전환 오버헤드(dispatch)"란?

이것이 이 보고서에서 가장 중요한 개념이다.

NPU가 AI 모델을 실행할 때, 모델을 **1,619개의 작은 단계(레이어)**로 나눠서 순서대로 실행한다. 각 단계를 시작할 때마다 NPU는:

1. "다음에 뭘 해야 하지?" (스케줄링)
2. "이 데이터를 어디서 가져오지?" (메모리 접근)
3. "결과를 어디에 저장하지?" (메모리 할당)

이 **준비 작업**에 매번 약 **19마이크로초(0.019ms)**가 걸린다.

```
하나의 레이어:  [준비 19µs] [실제 계산 ~1µs]
                 ~~~~~~~~~   ~~~~~~~~~~~~~~
                 95% 시간     5% 시간

1,619개 레이어: 1,619 × 19µs = 30.7ms ← 이게 전체 시간의 88%!
```

**비유로 설명하면:**
요리사가 1,619가지 재료를 볶아야 하는데, 각 재료를 볶는 데 1초 걸리고, 냉장고에서 재료를 꺼내오는 데 19초 걸린다. 실제 요리 시간은 27분(1,619초)인데, 재료 꺼내오는 시간만 8.5시간(30,761초)이 걸리는 것이다.

### 3.3 왜 이 문제가 소프트웨어로 해결이 안 되는가

이 "준비 작업" 시간은 **NPU 드라이버(커널 모듈)** 내부에서 발생한다. 드라이버는 운영체제의 가장 깊은 곳에서 돌아가는 프로그램이다. 우리가 수정할 수 있는 건 드라이버 "위"에서 돌아가는 프로그램뿐이다.

```
[우리가 수정 가능한 영역]
    ↕ Python 코드, ONNX 그래프, RKNN 설정
[경계선 ─────────────────────]
[수정 불가능한 영역]
    ↕ RKNN Runtime (librknnrt.so)  ← 바이너리, 소스 비공개
    ↕ NPU 커널 드라이버 (rknpu 0.9.6) ← 커널에 내장됨
    ↕ NPU 하드웨어
```

## 5. 12가지 최적화 시도 — 상세

### 시도 1: RKNN 빌드 설정 변경

**아이디어:** RKNN 변환기에는 여러 설정 옵션이 있다. 다른 옵션을 쓰면 더 빠른 모델이 나올 수도 있지 않을까?

**시도한 설정들:**

| 옵션 | 의미 | 결과 |
|------|------|------|
| `optimization_level=2` | 최적화 수준을 3(최대)에서 2로 낮춤 | 37.9ms (변화 없음) |
| `single_core_mode=True` | 단일 코어 전용 모델 생성 | 37.8ms (변화 없음) |
| `enable_flash_attention=True` | Flash Attention 최적화 활성화 | 37.4ms (변화 없음) |
| onnxsim 적용 | ONNX 모델 단순화 후 변환 | 37.6ms (변화 없음) |

**왜 안 됐나:** RKNN 변환기가 이미 내부적으로 최적화를 다 하고 있어서, 설정을 바꿔도 최종 결과물은 거의 동일했다. ONNX 노드를 5,818개에서 2,275개로 줄여도(-61%) RKNN 컴파일 결과는 같았다. 이는 RKNN이 자체 IR(중간 표현)로 변환하기 때문.

**결론:** ❌ 효과 없음

---

### 시도 2: NPU 3코어 동시 사용

**아이디어:** RK3588 NPU에는 3개의 코어가 있다. 한 코어로 못하면 세 코어를 다 쓰면 3배 빨라지지 않을까?

```
[Core 0] ─── 33.7ms
[Core 1] ─── 33.6ms
[Core 2] ─── 33.7ms
[Core 0+1+2] ─── 34.6ms  ← 오히려 느려짐!
```

**왜 안 됐나:** 병목이 "계산량"이 아니라 "레이어 전환 시간"이기 때문. 코어를 늘려도 레이어 전환 횟수는 동일(1,619회). 오히려 코어 간 동기화 오버헤드가 추가되어 약간 느려졌다.

**비유:** 한 사람이 냉장고에서 1,619가지 재료를 꺼내는 게 느린 건데, 요리사 3명을 고용해도 냉장고는 하나라서 빨라지지 않는 것과 같다.

**결론:** ❌ 효과 없음

---

### 시도 3: 배치 동기화 (Batch Sync) ✅ 유일한 성공

**아이디어:** NPU가 결과를 36개 출력 텐서에 쓰는데, 매번 하나씩 "결과 나왔어?" 하고 확인(sync)하지 말고, 36개 전부 한 번에 확인하면 어떨까?

**기존 방식 (개별 sync):**
```
출력1: sync → read → convert
출력2: sync → read → convert
...
출력36: sync → read → convert
→ 총 2.2ms
```

**새 방식 (배치 sync):**
```
sync1, sync2, ... sync36  (한 번에 동기화)
→ read1, convert1
→ read2, convert2
...
→ 총 1.2ms
```

**결과:** 캐시 처리 시간 **2.2ms → 1.2ms** (-1.0ms), 전체 **34.4ms → 32.7ms** (-1.7ms)

**왜 됐나:** 동기화(sync)는 NPU에게 "결과를 메인 메모리로 써줘"라고 요청하는 것이다. 하나씩 요청하면 매번 NPU가 응답을 기다려야 하지만, 한꺼번에 요청하면 NPU가 효율적으로 배치 처리할 수 있다.

**결론:** ✅ **-1.7ms 개선** (34.4ms → 32.7ms)

---

### 시도 4: ONNX 그래프 수술

**아이디어:** AI 모델 내부의 연산 구조를 직접 수정해서, 불필요한 연산을 줄이면 RKNN 변환 시 레이어 수가 줄어들지 않을까?

#### 4a. Pointwise Conv1D → MatMul 교체

Zipformer 모델 안에는 "pointwise convolution"이라는 연산이 60개 있다. 이 연산은 수학적으로 "행렬 곱셈(MatMul)"과 동일하지만, 데이터 배치 방향이 다르다. Conv1D는 데이터를 NCT(채널-먼저) 형태로 받고, MatMul은 NTC(시간-먼저) 형태로 받는다.

```
현재: [Transpose NCT→NTC] → [Conv1D] → [Transpose NTC→NCT]
목표: [MatMul]  (Transpose 2개 제거!)
```

30개 교체에 성공하고 60개 Transpose를 제거했지만, **중간에 있는 depthwise conv가 NCT 형식을 필요로 해서** 전체적인 데이터 흐름이 깨졌다. shape 불일치로 실행 불가.

#### 4b. Reshape-Transpose 체인 융합

연속된 `Reshape → Transpose` 패턴이 106개 있었다. 이걸 하나의 `Reshape`로 합치면 노드 수가 줄어든다. 하지만 Reshape(모양 변경)과 Transpose(축 교환)는 근본적으로 다른 연산이라 단순 합치기가 불가능했다.

**결론:** ❌ 레이아웃 불일치/융합 불가

---

### 시도 5: op_target — 특정 연산을 CPU에서 실행

**아이디어:** Reshape와 Transpose는 실제로 "데이터를 재배치"하는 연산이다. NPU에서 하면 dispatch 오버헤드가 들지만, CPU에서 하면 메모리 포인터만 바꾸면 되니까 거의 공짜다. 이런 연산을 CPU로 옮기면?

#### 5a. RKNN 내부 레이어 이름 발견

RKNN의 `op_target` 옵션은 특정 레이어를 CPU에서 실행하도록 지정할 수 있다. 하지만 **ONNX 노드 이름으로는 작동하지 않는다.** RKNN이 자체적으로 부여하는 내부 이름이 필요하다.

`accuracy_analysis()` 기능을 사용해서 1,619개 RKNN 내부 레이어 이름을 전부 추출했다:

```
예시:
  /encoder_embed/Reshape_output_0_tp-rs    (Transpose+Reshape 융합 레이어)
  /Add_output_0-rs_tp                       (Add 후 Reshape+Transpose)
  /encoder_embed/out/MatMul_output_0_mm_tp_sw  (MatMul+Transpose+Swish 융합)
```

접미사의 의미:
- `_rs` = Reshape
- `_tp` = Transpose
- `_mm` = MatMul
- `_sw` = Swish (활성화 함수)

#### 5b. 전체 Reshape/Transpose CPU 이동 → 크래시

265개의 모든 Reshape/Transpose 레이어를 CPU로 지정했다. 빌드는 성공했지만 실행 시 **Segfault(프로그램 충돌)**:

```
Unsupport CPU op: exMatMul
Segmentation fault
```

**원인:** Reshape를 CPU로 옮기면, 그 Reshape와 **융합(fused)**되어 있는 연산(exMatMul, exSwish 등)도 함께 CPU로 끌려간다. 이 융합 연산들은 NPU 전용으로, CPU에서는 실행할 수 없다.

```
NPU에서: [Reshape + MatMul + Transpose + Swish] → 하나의 융합 연산
CPU로 옮기면: Reshape(OK) + exMatMul(???) + exSwish(???) → 크래시!
```

**결론:** ❌ 크래시 (cascade 문제)

---

### 시도 6: FP16 (16비트 부동소수점) 빌드

**아이디어:** 현재 INT8(8비트 정수)로 양자화된 모델을 사용 중인데, INT8에서는 데이터 변환(DataConvert) 레이어가 62개 추가된다. FP16이면 이 레이어가 없어서 총 레이어 수가 줄어들 수 있다.

| 모델 | 추론 시간 | 크기 |
|------|----------|------|
| INT8 | 36.2ms | 79.6 MB |
| **FP16** | **46.2ms** | **147 MB** |

**왜 오히려 느려졌나:** FP16은 모든 연산을 16비트로 하므로, INT8보다 **계산량이 2배**다. DataConvert 레이어가 줄어드는 것보다 계산량 증가가 훨씬 크다. 모델 크기도 2배로 늘어서 메모리 전송도 더 걸린다.

**비유:** 간단한 메모(INT8)로 소통하던 걸 장문의 편지(FP16)로 바꾸면, 중간 번역(DataConvert)은 안 해도 되지만, 편지 쓰고 읽는 시간이 훨씬 오래 걸리는 것.

**결론:** ❌ +10ms 악화

### 시도 7: 선택적 op_target — 소수의 안전한 레이어만 CPU로

**아이디어:** 시도 5에서 전체를 옮기면 크래시가 나니까, "안전한" 레이어(융합 연산이 아닌 순수 Reshape/Transpose)만 골라서 소수만 CPU로 옮기면?

316개의 "안전한 후보"를 식별하고, 점진적으로 테스트:

| CPU로 이동한 레이어 수 | 추론 시간 | 변화 |
|---------------------|----------|------|
| 0 (baseline) | 36.3ms | — |
| 10개 | 35.4ms | **-0.9ms** |
| 50개 | 37.8ms | **+1.5ms (악화!)** |

**왜 50개는 오히려 느려졌나:** CPU에서 Reshape/Transpose를 실행하는 것 자체에도 시간이 든다. 10개 정도는 NPU dispatch 절감이 CPU 실행 시간을 이기지만, 50개가 되면 CPU 실행 시간이 더 커진다. 또한 NPU→CPU→NPU로 데이터가 왔다갔다하는 오버헤드도 있다.

**결론:** ❌ 미미한 개선 (-0.9ms), 스케일링 불가

### 시도 8: SenseVoice 모델 조사

**아이디어:** Zipformer 대신 다른 AI 모델을 쓰면 어떨까? SenseVoice라는 모델이 RK3588에서 20배 실시간 속도를 낸다고 한다.

**조사 결과:**

| 항목 | SenseVoice | Zipformer (현재) |
|------|-----------|-----------------|
| 파라미터 | 234M | 69M |
| 한국어 지원 | ✅ | ✅ |

### 시도 9: 모델 2분할 (분석만)

**아이디어:** 1,619개 레이어를 가진 하나의 큰 모델을, 약 800개씩 두 개의 모델로 나누면? 각 모델의 dispatch 시간이 줄어들 수 있다.

**분석 결과:**
```
현재:  [1,619 레이어 모델] → dispatch = 1,619 × 19µs = 31ms

분할:  [모델A: 800 레이어] → dispatch = 800 × 19µs = 15ms
       [모델B: 819 레이어] → dispatch = 819 × 19µs = 16ms
       총합 = 31ms + 모델 전환 오버헤드 = 31ms+α  ← 오히려 더 느림!
```

총 레이어 수가 줄지 않으므로 dispatch 총합은 동일하고, 모델 전환 오버헤드만 추가된다.

**결론:** ❌ 분석 단계에서 기각 (빌드 미시도)

### 시도 10: model_pruning + compress_weight + sparse_infer

**아이디어:** RKNN 변환기에 내장된 최적화 기능들을 활용해보자.

| 옵션 | 의미 | 결과 |
|------|------|------|
| `model_pruning=True` | 불필요한 노드를 컴파일러가 제거 | 35.1ms (노이즈 수준 변화) |
| `compress_weight=True` | 가중치 압축으로 메모리 전송 감소 | 35.1ms (위와 동일) |
| `sparse_infer=True` | 희소 행렬 최적화 | **rk3588 미지원** |
| `quantized_dtype='w8a16'` | 8비트 가중치 + 16비트 활성화 | **rk3588 미지원** |

**왜 안 됐나:**
- `model_pruning`: 이미 학습 시 불필요한 부분이 없는 모델이라, 컴파일러가 제거할 게 없었다.
- `compress_weight`: 가중치를 압축해도 NPU 내부에서 해제해야 하므로 속도에 영향이 거의 없었다.
- `sparse_infer`, `w8a16`: RK3588 NPU 하드웨어가 이 기능을 지원하지 않았다.

**결론:** ❌ 효과 없음 또는 미지원

### 시도 12: ONNX Shape 노드 상수 폴딩

**아이디어:** ONNX 모델에는 "이 텐서의 크기를 알아내는" 연산(Shape, Gather, Unsqueeze)이 401개나 있다. 우리는 입력 크기가 고정되어 있으니, 이 연산들을 미리 계산한 상수로 교체하면 노드 수가 크게 줄어든다.

```
원래: [Shape] → [Gather] → [Unsqueeze] → [Concat] → [Reshape에 shape 전달]
         (4개 노드)

교체 후: [상수 텐서] → [Reshape에 shape 전달]
         (0개 노드, 상수만 존재)
```

**결과:**

| 항목 | 값 |
|------|-----|
| 원본 ONNX | 2,275 노드 |
| **교체 후 ONNX** | **1,874 노드 (-401개, -17.6%)** |
| ORT(CPU) 검증 | ✅ 정상 작동 |
| RKNN 빌드 | ❌ **실패** |

```
오류: "All outputs ['encoder_out'] of model are constants, The model is invalid!"
```

**왜 실패했나:** RKNN 변환기의 constant folding(상수 전파) 단계에서, 우리가 추가한 상수들이 연쇄적으로 전파되어 최종 출력까지 "상수"로 판단해버렸다. ONNX 레벨에서는 정상이지만, RKNN의 내부 그래프 분석 로직이 원본 Shape 노드에 의존하고 있어서 호환이 안 됐다.

**핵심 교훈:** ONNX 노드 수와 RKNN 레이어 수는 1:1이 아니다. RKNN은 자체 IR로 변환하며, ONNX에서 노드를 줄여도 RKNN 레이어가 비례로 줄어들지 않는다.

**결론:** ❌ RKNN 호환 문제

**아이디어:** 현재 Python으로 NPU를 제어하고 있다. C++로 바꾸면 함수 호출 오버헤드가 줄지 않을까?

**결론:** ❌ 무의미 (0.1ms 절감)

## 7. 왜 20ms를 달성할 수 없는가

### 7.1 수학적 한계

```
현재:
  1,619 레이어 × 19µs/레이어 = 30.7ms (dispatch overhead)

20ms 달성을 위한 조건 (둘 중 하나):
  조건 A: 레이어 수를 1,050으로 감소 (-35%)
  조건 B: dispatch 시간을 12µs/레이어로 감소 (-37%)

두 조건 모두 소프트웨어 레벨에서 불가능:
  - 조건 A: 레이어 수는 모델 구조에 의해 결정. 변경하려면 모델 재학습 필요.
  - 조건 B: dispatch 시간은 NPU 커널 드라이버 내부 구현. 바이너리 수정 불가.
```

### 7.2 구조적 병목 다이어그램

```
                    ┌──────────────────────────────┐
                    │  우리가 제어할 수 있는 영역    │
                    │                              │
                    │  ✅ 입출력 데이터 처리        │ → 1.2ms (batch sync로 최적화 완료)
                    │  ✅ 캐시 변환 방식            │
                    │  ✅ ONNX 그래프 구조         │ → RKNN에 반영 안 됨
                    │  ✅ RKNN 빌드 설정           │ → 효과 없음
                    │                              │
                    └──────────┬───────────────────┘
                               │
                    ╔══════════╧═══════════════════╗
                    ║  우리가 제어할 수 없는 영역    ║
                    ║                              ║
                    ║  ⛔ RKNN Runtime 내부         ║ → dispatch 19µs/layer (고정)
                    ║  ⛔ NPU 커널 드라이버 0.9.6   ║ → 커널 빌트인, 교체 불가
                    ║  ⛔ NPU 하드웨어 설계         ║
                    ║                              ║
                    ╚══════════════════════════════╝
                               ↑
                          31ms의 병목이 여기에 있음
```

## 8. 향후 가능한 경로

### 8.1 NPU 커널 드라이버 업그레이드 (가장 유망)

| 항목 | 현재 | 목표 |
|------|------|------|
| 커널 드라이버 | 0.9.6 (2024-03-22) | 최신 (2025-xx) |
| Runtime | 2.3.2 (최신) | 동일 |
| 예상 효과 | dispatch ~19µs/layer | dispatch ~12µs/layer? |
| 예상 시간 | 31ms | ~21ms? |

**방법:** 커널 소스 코드에서 RKNPU 드라이버를 최신 버전으로 교체하고 커널 재빌드.
**난이도:** 중간 (리눅스 커널 빌드 경험 필요)
**위험:** 다른 드라이버와의 호환성 문제 가능

현재 커널 드라이버(0.9.6)는 Runtime(2.3.2)보다 **1년 이상 구형**이다. Rockchip이 새 드라이버에서 dispatch 효율을 개선했을 가능성이 높다.

### 8.2 더 작은 모델로 재학습

| 항목 | 현재 | 목표 |
|------|------|------|
| 파라미터 | 69M | ~20M |
| RKNN 레이어 | ~1,619 | ~500-800 |
| 예상 시간 | 31ms | ~10-16ms |
| CER | 26.25% | 30-40%? (악화 예상) |

**방법:** 더 작은 Zipformer 설정으로 한국어 데이터(KsponSpeech)에서 재학습.
**난이도:** 높음 (GPU 학습 인프라 + 한국어 데이터 필요)

### 8.3 RKNN MatMul API 하이브리드

**개념:** 전체 모델을 RKNN으로 돌리지 말고, 핵심 행렬 곱셈만 NPU에서 실행하고 나머지는 CPU에서 처리.

```
현재:  전체 모델 → NPU (1,619 레이어, 31ms dispatch)
하이브리드:
  Attention QKV 계산 → NPU (MatMul API, ~10 호출)
  Reshape/Transpose  → CPU (무시 가능한 시간)
  Feed-forward       → NPU (MatMul API, ~10 호출)
  → 총 NPU 호출: ~50회, dispatch = 50 × 19µs ≈ 1ms!
```

**난이도:** 매우 높음 (모델 전체를 수동으로 분해/재구성)
**예상 효과:** 미지수 (MatMul API의 개별 호출 오버헤드가 다를 수 있음)

## 10. 결론

### 달성한 것

- 12가지 서로 다른 최적화 방법을 체계적으로 시도하고 기록
- Batch sync 최적화로 -1.7ms 개선 (34.4ms → 32.7ms)
- RKNN 내부 레이어 이름 추출 방법 발견 (accuracy_analysis)
- ONNX Shape 상수 폴딩으로 노드 17.6% 감소 (RKNN 호환 문제로 미적용)
- NPU 병목의 정확한 원인 규명 (per-layer dispatch overhead)

### 달성하지 못한 것

- **20ms 목표 미달성** — 현재 ~35ms (avg)
- 근본 병목(dispatch overhead)은 소프트웨어 레벨에서 해결 불가

### 핵심 교훈

1. **NPU ≠ 항상 빠름.** NPU가 빠른 건 계산 자체(2ms)이지, 1,619번 반복되는 레이어 전환(31ms)이 전체 시간의 88%를 차지한다.
2. **Transformer 구조는 NPU에 불리하다.** 캐시 텐서가 많고(36개), multi-head attention의 head 분리/결합에 많은 Reshape/Transpose가 필요하다.
3. **ONNX 최적화 ≠ RKNN 최적화.** ONNX 노드를 줄여도 RKNN 내부 레이어는 비례로 줄지 않는다.
4. **커널 드라이버 업그레이드가 가장 유망한 경로.** 현재 드라이버(0.9.6)가 Runtime(2.3.2)보다 1년 이상 구형이며, dispatch 효율 개선이 기대된다.
